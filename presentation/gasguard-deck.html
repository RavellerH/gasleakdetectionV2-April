The Challenge
Gas leaks in refineries are
invisible until it's too late.
⚠ Safety Risk
Undetected hydrocarbon gas accumulation leads to explosions, fires, and toxic exposure — the leading cause of industrial fatalities in refinery environments.
⏱ Slow Response
Manual inspection rounds cannot provide continuous coverage. Hours may pass between rounds — a leak detected at hour 3 may have already reached critical concentration.
📋 Compliance Burden
Regulatory bodies require documented evidence of monitoring, incident response, and corrective actions. Manual paper trails are incomplete and hard to audit.
🗺 No Cross-Site Visibility
With multiple refinery units spread across a site, there is no unified view. Supervisors lack the ability to compare conditions across units or identify cross-site patterns.
🔋 Device Blind Spots
Battery failures and network disconnections in remote sensors go unnoticed, creating monitoring gaps exactly where and when coverage is most critical.
GASGUARD addresses all of these →
Continuous monitoring · Instant alerts · Device health tracking · Multi-site visibility · Automated audit trail

How GASGUARD is built — the hardware topology, data flow, and the technology stack that powers it.
01
Architecture
Three-tier architecture
Layer 1 — Field
Sensor Network
Wireless IoT mesh of gas sensors, cluster heads, and gateway nodes deployed across refinery units.
  • 28 MQ-series gas sensors
  • 22 cluster head relays
  • 7 gateway nodes (mesh root)
  • Star + mesh hybrid topology
Layer 2 — Backend
API & Data Layer
NestJS server exposes a GraphQL API. Prisma ORM manages all device data, readings, events, and user records.
  • NestJS API framework
  • GraphQL code-first schema
  • Prisma ORM
  • SQLite → upgradeable
Layer 3 — Frontend
Operator Dashboard
Next.js 15 React application. Real-time polling, interactive 3D map, analytics charts, and compliance reports.
  • Next.js 15 + React
  • Mapbox GL 3D mapping
  • Recharts analytics
  • graphql-request client
Network Design
Star + Mesh hybrid topology
🛜 Gateway (Mesh Root)
7 units — one or two per refinery unit. Forms the backbone mesh network. Forwards all aggregated data to the backend server via the primary uplink.
RSSI: −30 dBm avg Grade A
📡 Cluster Head (Relay)
22 units distributed per RU. Aggregates readings from nearby sensors over star topology and relays upstream to the gateway over mesh.
Dual RSSI: mesh + star
🔵 Sensor Node
28 MQ-series gas sensors. Reports ppm readings to its parent cluster head every 20 seconds. Battery-powered with health monitoring.
Battery: 60–90% SOC

Intelligent alert system
Alerts are automatically generated when sensors breach thresholds or devices change status. A 15-minute deduplication window prevents flood alerts from the same sensor.
Alert Types

Alert Panel Features

Smart Deduplication
When a sensor is continuously breaching threshold, only one event is logged per 15-minute window — keeping the alert queue actionable, not overwhelming.

Backend Processing
  • PPM reading stored in Prisma/SQLite

Frontend Polling

Technical — Security
Security & access control
Role-Based Access
ADMIN
Full access to all RUs. Can manage users, configure thresholds, view all events across all sites.
OPERATOR
Scoped to their assigned RU. Can view devices, acknowledge events, and export reports for their unit only.
Rate Limiting
  • Max 5 login attempts per account
  • 15-minute lockout window
  • In-memory rate limit map per email
Audit Trail
  • Every login/logout recorded with timestamp and operator email
  • All alert acknowledgements stored with mandatory operator note
  • Device state changes automatically logged
  • Complete chain from first detection → acknowledgement → resolution
Data Integrity
  • GraphQL input validation via class-validator
  • Prisma type-safe queries — no raw SQL injection risk
  • Acknowledgement notes validated non-empty before save
Dev mode: passwordless login for demonstration. Production mode adds bcrypt password hashing.
