Problem

Running all 6 RUs (RU2–RU7) simultaneously created cross-cutting concerns: shared firewall rules, mixed network zones, and regulatory boundaries that don't map cleanly to a single multi-tenant deployment.

Resolution

Platform scope narrowed to one RU per deployment instance. Each RU now runs as an isolated stack — its own network segment, firewall policy, and compliance boundary — eliminating the multi-RU security surface entirely.

RU-Specific Dashboard, Access Control & User Management

3 / 7

Delivered Checklist

  • ✓Dedicated dashboard scoped to the active RU's sensors & location
  • ✓Role-based access control (Admin / Operator / Viewer)
  • ✓Per-RU user management (add / remove / assign roles)
  • ✓Registration workflow tied to the RU
  • ✓Backend APIs filter all data by RU at the query layer

Success Criteria — All Met

Single RU has an isolated dashboard. Users only see and act on their assigned RU. Registration and management work end-to-end. Access is enforced at both UI and API levels. No cross-unit data leakage observed in verification testing.

Enforcement

RBAC is checked at both the UI (route guards, conditional rendering) and the API (middleware validates role + RU claim on every request). A user's session token carries a single RU id — requests for any other RU id are rejected with 403.
